# Workflow 开发指南

本指南记录在 workflow 设计讨论中沉淀的原则与决策。新原则需讨论确认后再写入，
避免单方面定调。按主题分类，不按时间。

---

## 一、核心设计哲学：必要元素为导向，而非硬编码

**原则**：workflow 应围绕"达成目标所需的必要元素"来设计，而不是为某个具体项目
硬编码调用约定。一个硬编码死的 workflow 换个项目就跑不了。

**推论**：
1. **不要假设统一的编码规范**。例如，不要假设所有项目的训练入口都接受 `--config`
   参数。`config` 对于一个项目可能是"模型结构声明"，对另一个可能根本不存在。
2. **获取必要元素，而非统一接口**。例如目标是训练不同模型，必要元素是"入口是哪个
   文件、基线模型是哪个、初始超参是什么、模型文件怎么传入"，workflow 应该让 agent
   自行查阅项目、做出判断、向用户确认这些元素，而不是强制所有项目走同一条命令行。
3. **给 agent 灵活度**，以目标为导向。workflow 可以有合理的假设限制（见下），但在
   假设范围内应让 agent 根据实际项目情况自主决策。

**合理的假设限制（可接受）**：
- 假设存在一个训练脚本入口。
- 假设入口能接收不同的模型文件作为输入，来训练不同模型。
- 假设存在 dummy input 可用作时延测量。

这些是 workflow 的适用前提，应该在 workflow 文档里显式声明，而不是藏在某个 agent
的实现里。

**反模式（要避免）**：
- 为单个 workflow 设计专用脚本然后塞进 harness 当"通用工具"。
- 把某个项目的调用约定（如固定的 `--config` schema）固化进 prompt。
- 过度设计：面向想象中的需求造一堆复杂机制，而非面向必要元素。

---

## 二、面向通用性设计 Harness，而非面向单个 workflow

**原则**：harness 提供的是**通用原子能力**，供任何 workflow 的 agent 调用。不要把
针对某个具体任务（如 NAS 训练监控）的脚本固化进 harness。

**对齐对象：Claude Code 的工具哲学**。Claude Code 提供 Bash、Read、Grep、Glob 等
通用原子工具，agent 用这些原子的组合完成任务，而不是每个任务一个专用工具。harness
应当如此。

**决策准则**：当考虑"要不要给 harness 加 X 能力"时，问自己——
> X 是一个通用原语（任何长任务 workflow 都需要），还是只是 NAS 的需求伪装成通用？

- 通用原语 → 加进 harness（如：后台任务的心跳/哨兵监控）。
- NAS 专用 → 留在 workflow 内（如：fitness 计算、候选树管理）。

---

## 三、Agent 间信息传递：上游尽可能把必要信息递给下游

**原则**：上游 agent 应该把下游完成任务所需的**必要信息**主动整理好传递，让下游
少读非必要文件。

**示例（非规范）**：上游 optimizer agent 不应只丢一句"去优化"，而应明确传递
"读取这个模型文件 → 按这个结构新建一个文件 → 做某方面优化"。下游因此不必重新遍历
项目结构、猜测入口。

**但**：传递的是"必要元素"，不是手把手指令。下游 agent 仍保有根据实际情况判断的
灵活度（见第一节）。

---

## 四、明确 agent 任务边界

**原则**：每个 agent 的任务必须明确单一，避免职责模糊导致的互相推诿或重复劳动。
"明确"不等于"死板"——agent 在明确边界内保有灵活度。

---

## 五、统一文件管理与断点续传

**原则**：workflow 应指定一个统一目录管理所有产物，每个 agent 都遵循相同的目录
约定，并支持断点续传（崩了重启能从已有产物恢复，不重跑已完成的步骤）。

---

## 六、长任务监控：组合现有原语，而非造专用工具

**原则**：长任务的"可靠完成 + 可见性"应靠 agent **组合通用原语**实现，而不是给
harness 造专用监控工具。

**确认的做法（对齐 Claude Code 实战）**：agent 自己写一个监控脚本，用 cron（或
等价的后台循环）定时执行，**定时摘取训练脚本被重定向后的输出（如 train.log）的
最新部分，判断训练是否正常运行**，并定期汇报进展；把运行中的 PID 记录在一个文件里，
可随时 kill。这套组合由现有原语构成：
- `bash run_in_background=true` 拉起训练，输出重定向到日志文件（harness 已有）
- agent 写的轮询/监控脚本（定时 `tail` 摘取最新输出，针对任务裁剪判断逻辑）
- cron / 后台循环（OS 原语，不是 harness 工具；不绑定具体调度器——sandbox 下 cron
  守护进程未必可靠，`while ... sleep` 后台循环更通用）
- PID 记录文件（agent 维护，读取即"列出任务"）

**关键约定**：
- 完成信号是**哨兵文件**（如 `status.json` 出现），不是进程退出——堵死"进程 exit 0
  但没产出结果"的假成功。
- 摘取的"什么算正常"由 agent 在脚本里现写，不硬编码（不同项目/不同训练，判断逻辑
  不同，如 loss 是否下降、有无 OOM、step 是否推进）。
- agent 主体不常驻轮询——轮询是后台脚本做的事，agent 只发起 + 最后收哨兵，避免 LLM
  常驻烧 token。

**结论**：
- 不造 `wait_for_tasks`（阻塞等待）专用工具——cron 模式的"定期汇报"对用户更有价值，
  且等待可由监控脚本写完成标记 + bash 轮询实现。
- 不造 `list_background_tasks`——agent 把已发起的任务记进文件，读文件即列表。
- harness 唯一值得做的微调：后台任务 ack 里**暴露 PID**（通用、2 行改动，便于 kill）。

## 七、评判以目标为准，不硬编码 fitness

**原则**：不要为 workflow 合成一个标量 fitness 公式。用户会为每个指标给出明确约束，
评判就按这些约束判断：达标则停，没达标则继续。

**推论**：
- 让 analyzer 按目标（多指标、各有约束）自行判断哪些变体"有潜力"。
- 不引入 fitness.py 之类的加权计算——权重是硬编码，违背"必要元素导向"。
- selector 从 analyzer 标记的"有潜力"架构中、结合方向多样性 + 树深度选下一轮基线。

## 八、合并职责以保上下文，不为拆分而拆分

**原则**：agent 拆分应服务于"明确单一职责"，而非机械地一刀切。当一个中间 agent 会
**丢失上下文**（拿不到上游刚生成的细节，被迫重读文件）时，应考虑把它的职责并回
上游。

**示例（NAS）**：变异完成后直接启动运行，不设独立 runner——因为 runner 不掌握变异
的上下文（改了什么、怎么跑），独立出来反而加重负担。生成变异的 agent 自己拉起训练
并监控，上下文不丢失。

## 九、实验记录分层级 + 汇总文件

**原则**：所有实验产物统一目录管理；既要每个实验的细粒度分析记录，也要一个汇总
文件供快速纵览。分层级存放，支持断点续传。

## 十、产物放 workflow 目录，不污染用户工作目录

**原则**：workflow 运行产物（生成的模型文件、记录、报告）应放在 workflow 自己的
session 目录下，用绝对路径引用，**不要写入用户的 working dir**。用户的 working dir
是他们的项目代码，应保持干净。session_id 可获取（init_session 机制），断点续传通过
复用已有 session 目录实现。
