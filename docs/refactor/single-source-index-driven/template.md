# 任务模板

> 复制此模板创建新任务。所有字段必填。

---

```markdown
# <Task ID>: <动词+宾语>

**Phase**: <P0|P1|P2a|P2b|P3|P4|P5|P6>
**预估**: ~<N> 分钟（≤ 20 分钟，超过就拆）
**依赖**: <任务 ID 列表，或 "无">
**状态**: ⬜ 未开始 / 🟡 进行中 / 🔴 阻塞 / 🟢 待 review / ✅ 已完成

## 功能点

<1-2 句话说明这个任务做什么、为什么需要。聚焦"做什么"，不展开怎么做。>

## 代码计划

具体到文件和函数级别。每个文件说明 add/change/delete。

- **`<file_path>`** (新增/修改/删除)
  - <具体改动 1>
  - <具体改动 2>

## 产出标准 (Definition of Done)

可观测、可验证的结果。不接受"代码写完了"作为完成。

- [ ] <可观测结果 1>
- [ ] <可观测结果 2>

## 验证方法

具体命令 + 预期输出。

```bash
# 自动化验证
<command>
# 预期：<result>

# 手动验证（如适用）
<steps>
```

## 代码质量检查

提交前自查：

- [ ] **单一职责**: <该函数/类只做一件事，描述>
- [ ] **开闭原则**: <扩展不需要修改现有代码，描述如何>
- [ ] **Fail loud**: <错误显式抛出或 log，不静默吞错>
- [ ] **无静默 except**: <所有 except 都有 logging 或 re-raise>
- [ ] **测试覆盖**: <新增逻辑有对应单测>
- [ ] **命名达意**: <无需注释即可理解>
- [ ] **无 magic number**: <常量提取或注释来源>

## Review 检查

> 完成后必须经过 review 才能 mark 为 ✅。Reviewer 关注**实现是否匹配计划**、**质量检查是否真的做了**、**是否有未覆盖的边界 case**。

- [ ] DoD 全部满足
- [ ] 所有测试通过
- [ ] 无回归（pre-existing failures 列表保持不变或减少）
- [ ] 质量自查 7 项全过
- [ ] Reviewer: <name>  Approved at <date>
```

---

## 拆分原则

1. **原子性**：每个任务可在 10-20 分钟内完成。超过就拆。
2. **独立性**：任务可独立提交、独立 review、独立回滚。
3. **可验证**：必须有具体的验证命令和预期输出。
4. **依赖明确**：必须声明依赖，避免漏改前置。
5. **不混合关注点**：一个任务要么纯后端，要么纯前端，要么纯测试。混合的话拆。
6. **不引入不必要的抽象**：不为"未来扩展"加空接口/空类。YAGNI。

## 状态机

```
⬜ 未开始 ──start──→ 🟡 进行中 ──done──→ 🟢 待 review ──approve──→ ✅ 已完成
                          │                                  │
                          └──block──→ 🔴 阻塞 ──unblock──────┘
                                       (记录阻塞原因)
```

更新 `TASKS.md` 主表的状态列。
